Project Goal and Purpose

The purpose of this project was to LEARN. I spent a lot of time researching and learning new technologies. I mainly wanted to build a full-stack application that had all the strings attached; log-in, sign-up, REST Api, database, third-party and browser-apis, Reactive Development, user profiles, and a trading functionality for a nice challenge. This project currently has around 10 active users, most of which are former batchmates from the coding bootcamp I attended.

Web Stack Choices

Java and Spring were chosen for the backend to provide an enterprise-level REST API. This provided a great approach to implement microservices with the modularity and scalability that Spring provides in synergy with Java. The Maven build tool was also another excellent addition, as it provided great management and packaging tooling for the server.

Angular is my framework of choice, so that was the immediate decision. It provided great view modularity through the various components I was able to build and route. Angular's RxJS state management was also instrumental in facilitating reactive development on the client-side and maintaining data streams. Design and styling were not the main concerns for this project, so I wanted something that would be quick to implement and package. I decided to go with Tailwind.

Problems and Thoughts

Along the way, the implementation of RxJS began to become more complex, so I needed to really take a step back and think about the design of the data streams I was constructing. This process involved learning more about Hot and Cold Observables, Higher Order Observables, and really digesting the flow of data between components.

One of the main features, that of trading Pokemon between users, was deceivingly more complicated than I had anticipated it to be. The reason being the many different unexpected behaviors that could take place during the interaction of two user accounts reactively. A lot of test cases needed to be implemented.

Naming conventions, naming conventions, naming conventions. There were many times thoughout this development cycle where I had to spend an unwanted amount of time trying to decipher zodiac code due to my poor naming conventions!

Lessons Learned

One of the most important takeaways from this project was to pay attention and take the time to properly name variables, objects, functions, etc... Not only will it make code easier to understand and read, but somebody new coming on to a codebase will have a much easier time understanding the thought process of other developers. I now make this a conscious concern and take more time to properly name things than I did in the past.

Furthermore, in the context of design, I discovered the vital importance of considering and thoroughly examining potential abnormal use cases. While an initial implementation of a particular functionality might seem straightforward, unexpected behaviors can undoubtedly arise. By contemplating the scalability and behavior of the functionality, adopting better design patterns can significantly improve the implementation timeline. Comprehensive testing also proved to be invaluable in such situations. I made sure to achieve over 90% code coverage through testing on my backend API, which effectively minimized potential time-consuming issues.
